# 85.9. Qualify

> [!Note]
> Qualifying leads and opportunities is crucial for effective sales and marketing strategies, as it ensures that resources are allocated to the most promising prospects.

  I'll create a comprehensive guide to explain **Qualify** in Snowflake using **pharmaceutical data** (claims, formulary, LAD, retail sales, 840/852/837 EDI transactions). In SQL, `QUALIFY` is a powerful clause that filters results based on **window functions**—like a `WHERE` clause specifically for ranked, numbered, or aggregated data within groups.

## What is QUALIFY in Snowflake?

The `QUALIFY` clause filters results **after** window functions (like `ROW_NUMBER()`, `RANK()`, `SUM() OVER`) have been calculated. Think of it as a gatekeeper that says "only show me records that meet specific ranking or aggregation criteria."

**Key Difference**:
- `WHERE` filters **raw rows** before processing
- `HAVING` filters **grouped aggregates** (GROUP BY results)  
- `QUALIFY` filters **window function results** (row-by-row rankings/aggregates)

## Use Case 1: Deduplicating Claims (837) - Most Recent per Patient/Drug

**Scenario**: Qualify only the most recent claim for each patient-drug combination, removing duplicates from resubmissions.

```sql
-- Create claims data with duplicates (resubmissions, corrections)
CREATE OR REPLACE TABLE claims_with_duplicates (
    claim_id VARCHAR(30),
    patient_id VARCHAR(30),
    ndc_code VARCHAR(11),
    service_date DATE,
    billed_amount DECIMAL(10,2),
    claim_status VARCHAR(20),
    submission_timestamp TIMESTAMP,
    is_correction BOOLEAN
);

-- Insert data with multiple versions of same claim
INSERT INTO claims_with_duplicates VALUES
('CLM_001_V1', 'PT_1001', '00093-1058-01', '2024-01-15', 45.00, 'denied', '2024-01-16 09:00:00', FALSE),
('CLM_001_V2', 'PT_1001', '00093-1058-01', '2024-01-15', 45.00, 'pending', '2024-01-17 10:30:00', TRUE),  -- Resubmission
('CLM_001_V3', 'PT_1001', '00093-1058-01', '2024-01-15', 42.50, 'paid', '2024-01-18 14:00:00', TRUE),      -- Corrected amount

('CLM_002_V1', 'PT_1002', '00093-2748-01', '2024-01-20', 120.00, 'paid', '2024-01-21 08:00:00', FALSE),    -- Only one version
('CLM_003_V1', 'PT_1003', '61755-001-01', '2024-02-01', 1850.00, 'denied', '2024-02-02 09:15:00', FALSE),
('CLM_003_V2', 'PT_1003', '61755-001-01', '2024-02-01', 1850.00, 'paid', '2024-02-03 11:00:00', TRUE);

-- QUALIFY QUERY: Keep only the most recent claim per patient-drug
SELECT 
    claim_id,
    patient_id,
    ndc_code,
    service_date,
    billed_amount,
    claim_status,
    submission_timestamp,
    is_correction,
    
    -- Window function: Rank by submission time (most recent = 1)
    ROW_NUMBER() OVER (
        PARTITION BY patient_id, ndc_code, service_date 
        ORDER BY submission_timestamp DESC
    ) AS claim_version_rank
    
FROM claims_with_duplicates

-- QUALIFY CLAUSE: Only keep rank #1 (most recent submission)
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY patient_id, ndc_code, service_date 
    ORDER BY submission_timestamp DESC
) = 1;

-- Alternative: Using a CTE for clarity (beginner-friendly approach)
WITH ranked_claims AS (
    SELECT 
        *,
        ROW_NUMBER() OVER (
            PARTITION BY patient_id, ndc_code, service_date 
            ORDER BY submission_timestamp DESC
        ) AS rn
    FROM claims_with_duplicates
)
SELECT * 
FROM ranked_claims 
WHERE rn = 1;  -- This WHERE works on the window function result
```

**Workflow Explanation**:

1. **Window Function**: `ROW_NUMBER()` assigns 1, 2, 3... to each claim for the same patient-drug-date combination, ordered by submission time (newest first)
2. **PARTITION BY**: Groups claims by patient+drug+date (defines "duplicates")
3. **ORDER BY**: Puts most recent submission as rank 1
4. **QUALIFY**: Filters the result set to only keep rows where the rank = 1 (most recent)
5. **Result**: Only 3 rows returned (CLM_001_V3, CLM_002_V1, CLM_003_V2) instead of 6

## Use Case 2: Qualifying Top 3 Drugs per Therapeutic Class (Formulary Analysis)

**Scenario**: Qualify the top 3 highest-cost drugs within each therapeutic class for targeted negotiation.

```sql
-- Create formulary data
CREATE OR REPLACE TABLE formulary_for_qualify (
    ndc_code VARCHAR(11),
    drug_name VARCHAR(100),
    therapeutic_class VARCHAR(50),
    manufacturer VARCHAR(50),
    annual_cost DECIMAL(12,2),
    volume_sold INTEGER
);

INSERT INTO formulary_for_qualify VALUES
-- ACE Inhibitors (3 drugs)
('00093-1058-01', 'Lisinopril 10mg', 'ACE Inhibitors', 'Teva', 450000.00, 1000000),
('00093-1058-02', 'Lisinopril 20mg', 'ACE Inhibitors', 'Teva', 380000.00, 800000),
('00556-001-01', 'Enalapril 5mg', 'ACE Inhibitors', 'Sandoz', 220000.00, 600000),

-- Statins (4 drugs)
('00555-0808-02', 'Atorvastatin 20mg', 'Statins', 'Pfizer', 1200000.00, 3500000),
('00555-0808-01', 'Atorvastatin 10mg', 'Statins', 'Pfizer', 950000.00, 2800000),
('00603-001-01', 'Simvastatin 20mg', 'Statins', 'Teva', 650000.00, 2200000),
('00781-001-01', 'Rosuvastatin 10mg', 'Statins', 'AstraZeneca', 1800000.00, 1200000),

-- Specialty (2 drugs)
('61755-001-01', 'Revlimid 5mg', 'Immunomodulators', 'BMS', 12500000.00, 15000),
('61755-001-02', 'Revlimid 10mg', 'Immunomodulators', 'BMS', 18000000.00, 15000);

-- QUALIFY QUERY: Top 3 by cost per therapeutic class
SELECT 
    therapeutic_class,
    drug_name,
    manufacturer,
    annual_cost,
    volume_sold,
    ROUND(annual_cost / volume_sold, 2) AS cost_per_unit,
    
    -- Rank within therapeutic class
    RANK() OVER (
        PARTITION BY therapeutic_class 
        ORDER BY annual_cost DESC
    ) AS cost_rank_in_class,
    
    -- Total class spend for context
    SUM(annual_cost) OVER (PARTITION BY therapeutic_class) AS total_class_spend
    
FROM formulary_for_qualify

-- QUALIFY: Only keep ranks 1, 2, or 3 within each class
QUALIFY RANK() OVER (
    PARTITION BY therapeutic_class 
    ORDER BY annual_cost DESC
) <= 3

ORDER BY therapeutic_class, cost_rank_in_class;
```

**Workflow Explanation**:

1. **PARTITION BY therapeutic_class**: Resets ranking for each drug category (ACE, Statins, etc.)
2. **RANK()**: Assigns 1, 2, 3... within each class by annual cost (handles ties)
3. **QUALIFY <= 3**: Filters to only top 3 per class (returns 3 ACE + 3 Statins + 2 Specialty = 8 rows)
4. **Why not WHERE?** WHERE can't reference window functions; QUALIFY can
5. **Business Value**: Identifies negotiation targets within each category, not just overall top drugs

## Use Case 3: Qualifying High-Risk Patients for Care Management (>5 Claims/Month)

**Scenario**: Qualify patients with more than 5 claims in any rolling 30-day window for care management intervention.

```sql

CREATE OR REPLACE TABLE patient_claims_dates (
    claim_id VARCHAR(30),
    patient_id VARCHAR(30),
    claim_date DATE,

);

-- Insert claim data showing clustering
INSERT INTO patient_claims_dates VALUES
-- PT_2001: 6 claims in January (high frequency - qualify)
('C_001', 'PT_2001', '2024-01-05', 'Drug A', 50.00),
('C_002', 'PT_2001', '2024-01-10', 'Drug B', 75.00),
('C_003', 'PT_2001', '2024-01-12', 'Drug C', 45.00),
('C_004', 'PT_2001', '2024-01-15', 'Drug D', 120.00),
('C_005', 'PT_2001', '2024-01-18', 'Drug E', 85.00),
('C_006', 'PT_2001', '2024-01-25', 'Drug F', 95.00),

-- PT_2002: 3 claims spread out (don't qualify)
('C_007', 'PT_2002', '2024-01-05', 'Drug A', 50.00),
('C_008', 'PT_2002', '2024-01-20', 'Drug B', 75.00),
('C_009', 'PT_2002', '2024-02-15', 'Drug C', 45.00),

-- PT_2003: 7 claims in February (qualify)
('C_010', 'PT_2003', '2024-02-01', 'Drug X', 150.00),
('C_011', 'PT_2003', '2024-02-02', 'Drug Y', 200.00),
('C_012', 'PT_2003', '2024-02-05', 'Drug Z', 175.00),
('C_013', 'PT_2003', '2024-02-08', 'Drug X', 150.00),
('C_014', 'PT_2003', '2024-02-10', 'Drug Y', 200.00),
('C_015', 'PT_2003', '2024-02-12', 'Drug Z', 175.00),
('C_016', 'PT_2003', '2024-02-15', 'Drug X', 150.00);

-- QUALIFY QUERY: Patients with >5 claims in any 30-day period
SELECT 
    patient_id,
    claim_date,
    claim_id,
    cost,
    
    -- Count claims in previous 30 days (inclusive)
    COUNT(*) OVER (
        PARTITION BY patient_id 
        ORDER BY claim_date 
        RANGE BETWEEN 29 PRECEDING AND CURRENT ROW
    ) AS claims_in_last_30_days,
    
    -- Sum cost in last 30 days
    SUM(cost) OVER (
        PARTITION BY patient_id 
        ORDER BY claim_date 
        RANGE BETWEEN 29 PRECEDING AND CURRENT ROW
    ) AS spend_in_last_30_days
    
FROM patient_claims_dates

-- QUALIFY: Only rows where the 30-day count exceeds 5
QUALIFY COUNT(*) OVER (
    PARTITION BY patient_id 
    ORDER BY claim_date 
    RANGE BETWEEN 29 PRECEDING AND CURRENT ROW
) > 5

ORDER BY patient_id, claim_date;
```

**Workflow Explanation**:

1. **RANGE BETWEEN 29 PRECEDING**: Creates a rolling 30-day window (frame) for each patient
2. **COUNT(*) OVER**: Counts how many claims exist in that rolling window at each claim date
3. **QUALIFY > 5**: Only keeps claim rows where the patient had more than 5 claims in that 30-day period
4. **Result**: Shows PT_2001's claims from Jan 18 onwards (when count hit 6), and PT_2003's claims from Feb 8 onwards
5. **Use Case**: Identifies super-utilizers for pharmacy benefit management interventions

## Use Case 4: Qualifying Primary Payer per Patient (Rank by Payment Amount)

**Scenario**: When patients have multiple payers, qualify the primary payer as the one paying the most.

```sql
-- Create multi-payer claims data
CREATE OR REPLACE TABLE multi_payer_claims (
    claim_id VARCHAR(30),
    patient_id VARCHAR(30),
    ndc_code VARCHAR(11),
    payer_id VARCHAR(20),
    payer_type VARCHAR(20),
    paid_amount DECIMAL(10,2),
    paid_date DATE
);

INSERT INTO multi_payer_claims VALUES
-- Claim 1: Patient has Medicare primary, Medicaid secondary
('CLM_101', 'PT_3001', '00093-1058-01', 'MEDICARE_A', 'Medicare', 35.00, '2024-01-15'),
('CLM_101_CO', 'PT_3001', '00093-1058-01', 'MEDICAID_STATE', 'Medicaid', 10.00, '2024-01-15'),

-- Claim 2: Commercial primary only
('CLM_102', 'PT_3002', '00093-2748-01', 'COMM_BLUESHIELD', 'Commercial', 85.00, '2024-01-16'),

-- Claim 3: Medicare + Commercial supplemental (Medicare pays more)
('CLM_103', 'PT_3003', '61755-001-01', 'MEDICARE_B', 'Medicare', 850.00, '2024-01-20'),
('CLM_103_SUPP', 'PT_3003', '61755-001-01', 'COMM_AETNA', 'Commercial', 150.00, '2024-01-20'),

-- Claim 4: Split claim - Commercial primary, Patient secondary
('CLM_104', 'PT_3004', '00555-0808-02', 'COMM_CIGNA', 'Commercial', 95.00, '2024-01-22'),
('CLM_104_PT', 'PT_3004', '00555-0808-02', 'PATIENT_CASH', 'Patient', 45.00, '2024-01-22');

-- QUALIFY QUERY: Identify primary payer (highest payment) per claim
SELECT 
    claim_id,
    patient_id,
    ndc_code,
    payer_id,
    payer_type,
    paid_amount,
    
    -- Rank payers by amount paid (1 = primary/highest)
    RANK() OVER (
        PARTITION BY claim_id 
        ORDER BY paid_amount DESC
    ) AS payer_payment_rank,
    
    -- Percentage of total claim paid by this payer
    paid_amount / SUM(paid_amount) OVER (PARTITION BY claim_id) * 100 AS pct_of_total_claim
    
FROM multi_payer_claims

-- QUALIFY: Only the primary payer (rank 1)
QUALIFY RANK() OVER (
    PARTITION BY claim_id 
    ORDER BY paid_amount DESC
) = 1

ORDER BY claim_id;
```

**Workflow Explanation**:

1. **PARTITION BY claim_id**: Groups payment records by claim (all payers for one claim together)
2. **RANK() by paid_amount DESC**: Ranks payers 1, 2, 3... by who paid the most
3. **QUALIFY = 1**: Keeps only the top-paying payer for each claim
4. **pct_of_total_claim**: Calculates what % of the total bill this primary payer covers
5. **Result**: One row per claim showing the primary payer, useful for attribution analysis

## Use Case 5: Qualifying Inventory Lots for Disposal (FIFO - First Expired)

**Scenario**: Qualify inventory lots that should be dispensed first (earliest expiration) to minimize waste.

```sql
-- Create lot inventory
CREATE OR REPLACE TABLE inventory_lots_qualify (
    lot_id VARCHAR(30),
    ndc_code VARCHAR(11),
    drug_name VARCHAR(100),
    lot_number VARCHAR(20),
    expiration_date DATE,
    quantity_on_hand INTEGER,
    warehouse_location VARCHAR(20)
);

INSERT INTO inventory_lots_qualify VALUES
('LOT_001', '00093-1058-01', 'Lisinopril', 'LOT_A_2023', '2025-06-01', 500, 'WH_EAST'),
('LOT_002', '00093-1058-01', 'Lisinopril', 'LOT_B_2023', '2025-09-01', 800, 'WH_EAST'),
('LOT_003', '00093-1058-01', 'Lisinopril', 'LOT_C_2024', '2026-03-01', 1200, 'WH_EAST'),
('LOT_004', '00093-2748-01', 'Metformin', 'LOT_M_2023', '2025-01-15', 200, 'WH_WEST'),  -- Expiring soon!
('LOT_005', '00093-2748-01', 'Metformin', 'LOT_M_2024', '2026-06-01', 1500, 'WH_WEST'),
('LOT_006', '61755-001-01', 'Revlimid', 'LOT_R_2024', '2025-12-01', 50, 'WH_EAST');

-- QUALIFY QUERY: Select the oldest lot (by expiration) per drug per warehouse for priority dispensing
SELECT 
    ndc_code,
    drug_name,
    lot_number,
    expiration_date,
    quantity_on_hand,
    warehouse_location,
    DATEDIFF(DAY, CURRENT_DATE(), expiration_date) AS days_until_expiry,
    
    -- Rank lots by expiration (oldest first) within each drug-location
    ROW_NUMBER() OVER (
        PARTITION BY ndc_code, warehouse_location 
        ORDER BY expiration_date ASC, lot_id ASC
    ) AS fifo_rank
    
FROM inventory_lots_qualify

-- QUALIFY: Only the oldest lot per drug per location (the one to dispense first)
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY ndc_code, warehouse_location 
    ORDER BY expiration_date ASC, lot_id ASC
) = 1

ORDER BY days_until_expiry;
```

**Workflow Explanation**:

1. **PARTITION BY ndc_code, warehouse_location**: Ranks lots separately for each drug at each location
2. **ORDER BY expiration_date ASC**: Puts soonest-to-expire lots as rank 1
3. **QUALIFY = 1**: Keeps only the oldest lot for each drug-location combination
4. **days_until_expiry**: Calculates urgency (negative = already expired)
5. **Result**: Pick list showing exactly which lot to pull first for each drug at each warehouse

## Use Case 6: Qualifying Transactions for Auto-Adjudication (Outlier Detection)

**Scenario**: Qualify claims for automatic processing (straight-through processing) if they fall within normal parameters.

```sql
-- Create claims with historical context for outlier detection
CREATE OR REPLACE TABLE claims_for_adjudication (
    claim_id VARCHAR(30),
    ndc_code VARCHAR(11),
    pharmacy_npi VARCHAR(10),
    billed_amount DECIMAL(10,2),
    quantity_dispensed INTEGER,
    days_supply INTEGER,
    patient_zip VARCHAR(5)
);

-- Insert claims with one obvious outlier
INSERT INTO claims_for_adjudication VALUES
('CLM_201', '00093-1058-01', 'PHARM001', 45.00, 30, 30, '90210'),
('CLM_202', '00093-1058-01', 'PHARM001', 47.50, 30, 30, '90210'),
('CLM_203', '00093-1058-01', 'PHARM001', 44.00, 30, 30, '90210'),
('CLM_204', '00093-1058-01', 'PHARM001', 1350.00, 30, 30, '90210'),  -- Outlier! 30x normal
('CLM_205', '00093-1058-01', 'PHARM002', 46.00, 30, 30, '75201'),
('CLM_206', '00093-2748-01', 'PHARM001', 75.00, 60, 30, '90210'),
('CLM_207', '00093-2748-01', 'PHARM001', 1500.00, 60, 30, '90210');  -- Outlier!

-- QUALIFY QUERY: Identify claims within 2 standard deviations (qualified for auto-adjudication)
SELECT 
    claim_id,
    ndc_code,
    pharmacy_npi,
    billed_amount,
    quantity_dispensed,
    
    -- Statistical window functions
    AVG(billed_amount) OVER (PARTITION BY ndc_code) AS avg_amount_for_drug,
    STDDEV(billed_amount) OVER (PARTITION BY ndc_code) AS stddev_for_drug,
    
    -- Z-score (how many std devs from mean)
    (billed_amount - AVG(billed_amount) OVER (PARTITION BY ndc_code)) / 
        NULLIF(STDDEV(billed_amount) OVER (PARTITION BY ndc_code), 0) AS z_score,
    
    -- Check if within 2 std devs
    ABS((billed_amount - AVG(billed_amount) OVER (PARTITION BY ndc_code)) / 
        NULLIF(STDDEV(billed_amount) OVER (PARTITION BY ndc_code), 0)) AS abs_z_score
    
FROM claims_for_adjudication

-- QUALIFY: Only claims within 2 standard deviations (|z-score| <= 2)
QUALIFY ABS((billed_amount - AVG(billed_amount) OVER (PARTITION BY ndc_code)) / 
        NULLIF(STDDEV(billed_amount) OVER (PARTITION BY ndc_code), 0)) <= 2

ORDER BY ndc_code, billed_amount;
```

**Workflow Explanation**:

1. **AVG() OVER / STDDEV() OVER**: Calculates mean and standard deviation for each drug across all claims
2. **Z-score calculation**: Measures how unusual each claim is (0 = average, 3 = very unusual)
3. **QUALIFY <= 2**: Filters to only "normal" claims within 2 standard deviations (~95% of data)
4. **Result**: Excludes CLM_204 and CLM_207 (the outliers) from auto-adjudication, flagging them for manual review
5. **Business Rule**: Prevents fraudulent or erroneous claims from auto-paying

## Use Case 7: Multi-Criteria Lead Qualification (BANT Scoring for Prescribers)

**Scenario**: Qualify prescribers as "sales leads" using BANT-like criteria (Budget, Authority, Need, Timeline) based on claims data.

```sql
-- Create prescriber profile data
CREATE OR REPLACE TABLE prescriber_leads (
    prescriber_npi VARCHAR(10),
    prescriber_name VARCHAR(100),
    specialty VARCHAR(50),
    practice_state VARCHAR(2),
    
    -- BANT criteria
    annual_prescription_volume INTEGER,  -- Budget proxy
    high_cost_drugs_prescribed INTEGER,  -- Authority/Complexity
    unique_patients INTEGER,             -- Need/Reach
    days_since_last_rx INTEGER,          -- Timeline (recency)
    
    -- Additional factors
    ehr_system VARCHAR(50),
    prior_authorizations_submitted INTEGER
);

INSERT INTO prescriber_leads VALUES
('DR_001', 'Dr. Smith', 'Oncology', 'CA', 450, 120, 180, 2, 'Epic', 85),
('DR_002', 'Dr. Jones', 'Cardiology', 'NY', 1200, 45, 800, 5, 'Cerner', 25),
('DR_003', 'Dr. Lee', 'Family Practice', 'TX', 3500, 12, 2200, 1, 'Allscripts', 5),
('DR_004', 'Dr. Garcia', 'Oncology', 'FL', 380, 95, 150, 15, 'Epic', 60),
('DR_005', 'Dr. Patel', 'Endocrinology', 'IL', 890, 78, 420, 3, 'Athena', 40),
('DR_006', 'Dr. Chen', 'Family Practice', 'CA', 4200, 8, 2800, 0, 'Epic', 3);

-- QUALIFY QUERY: Qualify prescribers with composite score > 75
SELECT 
    prescriber_npi,
    prescriber_name,
    specialty,
    
    -- Individual component scores (0-100)
    LEAST(annual_prescription_volume / 50.0, 100) AS volume_score,
    LEAST(high_cost_drugs_prescribed * 1.0, 100) AS complexity_score,
    LEAST(unique_patients / 20.0, 100) AS reach_score,
    CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END AS recency_score,
    
    -- Composite BANT score
    (LEAST(annual_prescription_volume / 50.0, 100) * 0.3 +  -- Budget: 30%
     LEAST(high_cost_drugs_prescribed * 1.0, 100) * 0.25 + -- Authority: 25%
     LEAST(unique_patients / 20.0, 100) * 0.25 +           -- Need: 25%
     CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END * 0.2  -- Timeline: 20%
    ) AS bant_qualification_score,
    
    -- Rank by score
    RANK() OVER (ORDER BY 
        (LEAST(annual_prescription_volume / 50.0, 100) * 0.3 + 
         LEAST(high_cost_drugs_prescribed * 1.0, 100) * 0.25 + 
         LEAST(unique_patients / 20.0, 100) * 0.25 + 
         CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END * 0.2) 
        DESC
    ) AS lead_rank,
    
    -- Classification
    CASE 
        WHEN (LEAST(annual_prescription_volume / 50.0, 100) * 0.3 + 
              LEAST(high_cost_drugs_prescribed * 1.0, 100) * 0.25 + 
              LEAST(unique_patients / 20.0, 100) * 0.25 + 
              CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END * 0.2) >= 80 THEN 'HOT_LEAD'
        WHEN (LEAST(annual_prescription_volume / 50.0, 100) * 0.3 + 
              LEAST(high_cost_drugs_prescribed * 1.0, 100) * 0.25 + 
              LEAST(unique_patients / 20.0, 100) * 0.25 + 
              CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END * 0.2) >= 60 THEN 'WARM_LEAD'
        ELSE 'COLD_LEAD'
    END AS lead_temperature
    
FROM prescriber_leads

-- QUALIFY: Only qualified leads (score > 60)
QUALIFY (LEAST(annual_prescription_volume / 50.0, 100) * 0.3 + 
         LEAST(high_cost_drugs_prescribed * 1.0, 100) * 0.25 + 
         LEAST(unique_patients / 20.0, 100) * 0.25 + 
         CASE WHEN days_since_last_rx <= 7 THEN 100 ELSE 100 - days_since_last_rx END * 0.2) > 60

ORDER BY bant_qualification_score DESC;
```

**Workflow Explanation**:

1. **Component Scoring**: Converts raw metrics (volume, complexity) to 0-100 scale
2. **Weighted Average**: Applies BANT weights (Budget 30%, Authority 25%, Need 25%, Timeline 20%)
3. **QUALIFY > 60**: Filters to only "qualified" leads above threshold (removes DR_003 and DR_006)
4. **lead_temperature**: Categorizes into Hot/Warm/Cold for sales prioritization
5. **RANK()**: Orders qualified leads for sales team follow-up sequence

## Key QUALIFY Concepts Summary

| Concept | SQL Syntax | Business Meaning | Pharma Example |
|---------|-----------|------------------|----------------|
| **Deduplication** | `QUALIFY ROW_NUMBER() = 1` | Most recent/accurate record | Latest claim version |
| **Top N per Group** | `QUALIFY RANK() <= N` | Best performers per category | Top 3 drugs per class |
| **Threshold Filtering** | `QUALIFY COUNT(*) > X` | High-frequency detection | Super-utilizers |
| **Primary Selection** | `QUALIFY RANK() = 1` | Winner takes all | Primary payer |
| **Outlier Removal** | `QUALIFY ABS(z_score) <= 2` | Normal range only | Auto-adjudication |
| **Scoring** | `QUALIFY weighted_score > threshold` | Qualified leads | Sales targets |

**QUALIFY vs WHERE vs HAVING**:

```
Data Flow:
RAW DATA → WHERE (filter rows) → GROUP BY → HAVING (filter groups) 
→ WINDOW FUNCTIONS → QUALIFY (filter window results) → SELECT
```

**When to Use QUALIFY**:
- When filtering on `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`
- When filtering on `SUM() OVER`, `AVG() OVER` (running totals)
- When you need "top N per group" logic
- When deduplicating based on ordered criteria

**Remember**: Just like sales qualification ensures you focus on the best opportunities, SQL `QUALIFY` ensures you focus on the best rows—whether that's the most recent claim, the primary payer, or the highest-value drug in each category!

> [!Tip]
> Implementing a clear and consistent qualification framework can significantly improve the efficiency and effectiveness of sales and marketing efforts.

- Qualification is an ongoing process that requires continuous assessment and adjustment throughout the sales cycle.

- Step 3: Continuously review and refine the qualification process to ensure it remains effective and aligned with business objectives.

## Observe
- A common mistake people make is to rely too heavily on intuition or personal opinion when qualifying leads, rather than using a data-driven approach.
- A silent failure mode is when the qualification process becomes outdated or ineffective, leading to a decline in sales performance without obvious causes.
- A misleading assumption is that all leads and opportunities are created equal, and that qualification is not necessary or can be skipped.

> [!Important]
> For example, a company selling enterprise software might use a qualification framework that includes criteria such as company size, industry, and job function to identify high-potential leads, and then use data analysis and sales team feedback to continuously refine and improve the qualification process.
