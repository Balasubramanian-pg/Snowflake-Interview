# 85.1. Tumbling

> [!Note]
> Tumbling is a dynamic process involving the rotation of an object around a central axis, which can be applied in various fields such as gymnastics, engineering, and animation to achieve specific goals or effects.

## What is Tumbling in Data Engineering?

In data processing, **Tumbling Windows** are fixed-size, non-overlapping time intervals used to aggregate streaming data. Think of it like a gymnast's tumbling pass—each "tumble" (window) is a complete, distinct sequence with a clear start and end.

---

## Use Case 1: Gymnastics Competition Scoring System

**Scenario**: Track gymnast performances in 30-second tumbling windows during competitions.

```sql
-- Create sample data: Gymnast performance events
CREATE OR REPLACE TABLE gymnastics_events (
    event_id NUMBER,
    gymnast_name VARCHAR(50),
    skill_type VARCHAR(30),
    execution_score DECIMAL(4,2),
    difficulty_score DECIMAL(4,2),
    event_timestamp TIMESTAMP
);

-- Insert sample data simulating a competition
INSERT INTO gymnastics_events VALUES
(1, 'Simone Biles', 'Round-off', 9.5, 2.0, '2024-08-01 14:00:00'),
(2, 'Simone Biles', 'Back Handspring', 9.7, 2.5, '2024-08-01 14:00:15'),
(3, 'Simone Biles', 'Layout', 9.8, 3.0, '2024-08-01 14:00:28'),
(4, 'Jade Carey', 'Round-off', 9.3, 2.0, '2024-08-01 14:00:05'),
(5, 'Jade Carey', 'Back Handspring', 9.4, 2.5, '2024-08-01 14:00:18'),
(6, 'Jade Carey', 'Double Layout', 9.6, 3.5, '2024-08-01 14:00:32');

-- TUMBLING WINDOW QUERY: Aggregate scores in 30-second windows
SELECT 
    -- Step 1: Create tumbling windows using DATE_TRUNC
    DATE_TRUNC('MINUTE', event_timestamp) AS window_start,
    
    -- Step 2: Calculate window end (start + 30 seconds for this example)
    DATEADD(SECOND, 30, DATE_TRUNC('MINUTE', event_timestamp)) AS window_end,
    
    -- Step 3: Aggregate data within each window
    gymnast_name,
    COUNT(*) AS skills_performed,
    AVG(execution_score) AS avg_execution,
    AVG(difficulty_score) AS avg_difficulty,
    SUM(execution_score + difficulty_score) AS total_score
    
FROM gymnastics_events

-- Step 4: Group by the tumbling window and gymnast
GROUP BY 
    DATE_TRUNC('MINUTE', event_timestamp),
    gymnast_name

ORDER BY window_start, total_score DESC;
```

**Workflow Explanation**:
1. **DATE_TRUNC('MINUTE', timestamp)** - Cuts off seconds, creating fixed 1-minute windows (like a gymnast starting a new pass)
2. **DATEADD** - Defines the window boundary (start to end)
3. **GROUP BY** - Each window is independent (non-overlapping, like distinct tumbling passes)
4. **Aggregation** - Calculates performance metrics within that specific window only

## Use Case 2: Roller Coaster Safety Monitoring

**Scenario**: Monitor G-forces in 10-second tumbling windows for safety compliance.

```sql
-- Create sensor data table
CREATE OR REPLACE TABLE coaster_sensors (
    ride_id VARCHAR(20),
    sensor_location VARCHAR(30),
    g_force DECIMAL(5,2),
    rotation_speed DECIMAL(5,2),
    recorded_at TIMESTAMP
);

INSERT INTO coaster_sensors VALUES
('COASTER_01', 'Loop Entry', 4.5, 45.0, '2024-08-01 10:05:02'),
('COASTER_01', 'Corkscrew', 3.8, 60.0, '2024-08-01 10:05:08'),
('COASTER_01', 'Loop Exit', 4.2, 40.0, '2024-08-01 10:05:15'),
('COASTER_01', 'Helix', 3.5, 55.0, '2024-08-01 10:05:22'),
('COASTER_02', 'Loop Entry', 4.8, 48.0, '2024-08-01 10:05:03');

-- TUMBLING WINDOW: Safety analysis every 10 seconds
SELECT 
    -- Create 10-second tumbling windows
    TIME_SLICE(recorded_at, 10, 'SECOND', 'START') AS window_start,
    TIME_SLICE(recorded_at, 10, 'SECOND', 'END') AS window_end,
    
    ride_id,
    
    -- Safety metrics (tumbling = analyzing each "tumble" independently)
    MAX(g_force) AS peak_g_force,
    AVG(rotation_speed) AS avg_rotation,
    COUNT(*) AS sensor_readings,
    
    -- Safety flag based on tumbling window analysis
    CASE 
        WHEN MAX(g_force) > 5.0 THEN 'DANGER'
        WHEN MAX(g_force) > 4.5 THEN 'WARNING'
        ELSE 'SAFE'
    END AS safety_status
    
FROM coaster_sensors

GROUP BY 
    TIME_SLICE(recorded_at, 10, 'SECOND', 'START'),
    TIME_SLICE(recorded_at, 10, 'SECOND', 'END'),
    ride_id

ORDER BY window_start, ride_id;
```

**Workflow Explanation**:
1. **TIME_SLICE** - Snowflake's built-in tumbling window function (like slicing a gymnast's routine into segments)
2. **'START'/'END'** - Clearly defines non-overlapping boundaries
3. **Safety Analysis** - Each window is evaluated independently—danger in one window doesn't affect the next
4. **Non-overlapping** - 10:05:00-10:05:10 is completely separate from 10:05:10-10:05:20

## Use Case 3: Animation Frame Rendering Performance

**Scenario**: Analyze rendering times in 5-second tumbling windows for animation software.

```sql
CREATE OR REPLACE TABLE animation_frames (
    project_id VARCHAR(30),
    frame_number INTEGER,
    render_time_ms INTEGER,
    complexity_score INTEGER,
    rendered_at TIMESTAMP
);

INSERT INTO animation_frames VALUES
('PROJECT_TUMBLE', 1, 120, 5, '2024-08-01 09:00:01'),
('PROJECT_TUMBLE', 2, 135, 6, '2024-08-01 09:00:03'),
('PROJECT_TUMBLE', 3, 450, 9, '2024-08-01 09:00:06'), -- Complex tumbling animation
('PROJECT_TUMBLE', 4, 480, 9, '2024-08-01 09:00:08'),
('PROJECT_TUMBLE', 5, 150, 5, '2024-08-01 09:00:11'),
('PROJECT_FLIP', 1, 200, 7, '2024-08-01 09:00:02');

-- TUMBLING WINDOW: Performance analysis
WITH tumbling_windows AS (
    SELECT 
        -- Define 5-second tumbling windows
        FLOOR(DATEDIFF(MILLISECOND, '1970-01-01', rendered_at) / 5000) AS window_id,
        DATEADD(MILLISECOND, 
            FLOOR(DATEDIFF(MILLISECOND, '1970-01-01', rendered_at) / 5000) * 5000, 
            '1970-01-01') AS window_start,
        DATEADD(MILLISECOND, 
            (FLOOR(DATEDIFF(MILLISECOND, '1970-01-01', rendered_at) / 5000) + 1) * 5000, 
            '1970-01-01') AS window_end,
        
        project_id,
        frame_number,
        render_time_ms,
        complexity_score
        
    FROM animation_frames
)
SELECT 
    window_start,
    window_end,
    project_id,
    
    -- Performance metrics within each tumbling window
    COUNT(*) AS frames_rendered,
    AVG(render_time_ms) AS avg_render_time,
    MAX(render_time_ms) AS max_render_time,
    SUM(render_time_ms) AS total_window_time,
    
    -- Efficiency calculation (like controlling rotation axis in tumbling)
    AVG(render_time_ms * 1.0 / complexity_score) AS time_per_complexity_unit
    
FROM tumbling_windows

GROUP BY window_start, window_end, project_id

ORDER BY window_start, project_id;
```

**Workflow Explanation**:
1. **Convert to Epoch** - Changes timestamp to milliseconds since 1970
2. **FLOOR Division** - Groups time into fixed 5000ms (5-second) buckets (like fixed-size tumbling passes)
3. **Convert Back** - Transforms window IDs back to readable timestamps
4. **Independent Analysis** - Each 5-second window is analyzed separately—no overlap between windows

## Use Case 4: E-commerce Flash Sale Monitoring

**Scenario**: Track purchase velocity in 1-minute tumbling windows during flash sales.

```sql
CREATE OR REPLACE TABLE flash_sale_events (
    sale_id VARCHAR(20),
    product_id VARCHAR(20),
    user_id VARCHAR(30),
    action VARCHAR(20), -- 'view', 'cart', 'purchase'
    amount DECIMAL(10,2),
    event_time TIMESTAMP
);

INSERT INTO flash_sale_events VALUES
('SALE_001', 'GYM_SHOES', 'user_1', 'view', 0, '2024-08-01 12:00:05'),
('SALE_001', 'GYM_SHOES', 'user_2', 'cart', 89.99, '2024-08-01 12:00:15'),
('SALE_001', 'GYM_SHOES', 'user_3', 'purchase', 89.99, '2024-08-01 12:00:28'),
('SALE_001', 'YOGA_MAT', 'user_4', 'purchase', 45.00, '2024-08-01 12:00:45'),
('SALE_001', 'GYM_SHOES', 'user_5', 'purchase', 89.99, '2024-08-01 12:01:05'), -- Next window
('SALE_001', 'RESISTANCE_BANDS', 'user_6', 'purchase', 25.00, '2024-08-01 12:01:15');

-- TUMBLING WINDOW: Sales velocity analysis
SELECT 
    -- 1-minute tumbling windows (non-overlapping, distinct like tumbling passes)
    DATE_TRUNC('MINUTE', event_time) AS minute_window,
    
    sale_id,
    
    -- Conversion funnel within each tumbling window
    COUNT(CASE WHEN action = 'view' THEN 1 END) AS total_views,
    COUNT(CASE WHEN action = 'cart' THEN 1 END) AS total_carts,
    COUNT(CASE WHEN action = 'purchase' THEN 1 END) AS total_purchases,
    
    SUM(CASE WHEN action = 'purchase' THEN amount ELSE 0 END) AS revenue,
    
    -- Conversion rates (control metrics like controlling rotation axis)
    CASE 
        WHEN COUNT(CASE WHEN action = 'view' THEN 1 END) > 0 
        THEN COUNT(CASE WHEN action = 'purchase' THEN 1 END) * 100.0 / 
             COUNT(CASE WHEN action = 'view' THEN 1 END)
        ELSE 0 
    END AS conversion_rate_percent
    
FROM flash_sale_events

WHERE action IN ('view', 'cart', 'purchase')

GROUP BY 
    DATE_TRUNC('MINUTE', event_time),
    sale_id

ORDER BY minute_window;
```

**Workflow Explanation**:
1. **DATE_TRUNC('MINUTE')** - Creates fixed 1-minute windows (12:00:00-12:01:00, 12:01:00-12:02:00)
2. **Non-overlapping** - Events at 12:00:59 and 12:01:01 are in different windows
3. **Funnel Analysis** - Each window shows complete conversion cycle independently
4. **Control Metrics** - Conversion rate acts like "axis control"—measures stability of sales process

## Use Case 5: IoT Device Rotation Monitoring

**Scenario**: Monitor device orientation changes in 15-second tumbling windows.

```sql
CREATE OR REPLACE TABLE device_rotation (
    device_id VARCHAR(30),
    axis_x DECIMAL(8,5),
    axis_y DECIMAL(8,5),
    axis_z DECIMAL(8,5),
    rotation_speed DECIMAL(8,2),
    recorded_at TIMESTAMP
);

INSERT INTO device_rotation VALUES
('DEVICE_GYM_01', 0.5, 0.3, 0.8, 45.0, '2024-08-01 15:30:05'),
('DEVICE_GYM_01', 0.6, 0.4, 0.7, 60.0, '2024-08-01 15:30:12'),
('DEVICE_GYM_01', 0.4, 0.5, 0.9, 55.0, '2024-08-01 15:30:18'),
('DEVICE_GYM_01', 0.3, 0.6, 0.8, 50.0, '2024-08-01 15:30:25'),
('DEVICE_GYM_02', 0.2, 0.2, 0.95, 30.0, '2024-08-01 15:30:08');

-- TUMBLING WINDOW: Rotation stability analysis
SELECT 
    -- 15-second tumbling windows (like distinct tumbling sequences)
    TIME_SLICE(recorded_at, 15, 'SECOND', 'START') AS rotation_window_start,
    
    device_id,
    
    -- Axis stability metrics (core tumbling concept: controlling rotation axis)
    STDDEV(axis_x) AS x_axis_stability,  -- Lower = more stable
    STDDEV(axis_y) AS y_axis_stability,
    STDDEV(axis_z) AS z_axis_stability,
    
    -- Overall rotation control
    AVG(rotation_speed) AS avg_rotation_speed,
    MAX(rotation_speed) - MIN(rotation_speed) AS speed_variation,
    
    -- Stability score (like gymnast's form score)
    CASE 
        WHEN STDDEV(axis_x) < 0.1 AND STDDEV(axis_y) < 0.1 
        THEN 'EXCELLENT_CONTROL'
        WHEN STDDEV(axis_x) < 0.2 AND STDDEV(axis_y) < 0.2 
        THEN 'GOOD_CONTROL'
        ELSE 'NEEDS_IMPROVEMENT'
    END AS axis_control_rating
    
FROM device_rotation

GROUP BY 
    TIME_SLICE(recorded_at, 15, 'SECOND', 'START'),
    device_id

ORDER BY rotation_window_start, device_id;
```

**Workflow Explanation**:
1. **TIME_SLICE with 15 seconds** - Creates distinct, non-overlapping analysis periods
2. **Standard Deviation** - Measures axis stability (key tumbling principle: maintaining control over rotation axis)
3. **Stability Scoring** - Evaluates each window independently—unstable rotation in one window doesn't affect next
4. **Fixed Boundaries** - 15:30:00-15:30:15 is completely separate from 15:30:15-15:30:30

## Use Case 6: Social Media Trend Detection

**Scenario**: Detect hashtag velocity in 5-minute tumbling windows.

```sql
CREATE OR REPLACE TABLE social_posts (
    post_id VARCHAR(30),
    hashtag VARCHAR(50),
    engagement_score INTEGER,
    posted_at TIMESTAMP
);

INSERT INTO social_posts VALUES
('post_1', '#TumblingChamps', 150, '2024-08-01 18:00:05'),
('post_2', '#TumblingChamps', 230, '2024-08-01 18:02:15'),
('post_3', '#TumblingChamps', 450, '2024-08-01 18:04:45'),
('post_4', '#GymnasticsLife', 120, '2024-08-01 18:01:10'),
('post_5', '#TumblingChamps', 180, '2024-08-01 18:06:05'), -- Next window
('post_6', '#TumblingChamps', 300, '2024-08-01 18:08:20');

-- TUMBLING WINDOW: Trend velocity (like measuring tumbling pass momentum)
SELECT 
    -- 5-minute tumbling windows (distinct, complete cycles)
    DATE_TRUNC('HOUR', posted_at) AS hour_start,
    FLOOR(MINUTE(posted_at) / 5) * 5 AS minute_segment,
    
    -- Create readable window label
    DATEADD(MINUTE, FLOOR(MINUTE(posted_at) / 5) * 5, 
        DATE_TRUNC('HOUR', posted_at)) AS trend_window,
    
    hashtag,
    
    -- Velocity metrics (speed and direction of trend)
    COUNT(*) AS post_count,
    SUM(engagement_score) AS total_engagement,
    AVG(engagement_score) AS avg_engagement,
    
    -- Momentum calculation (like tumbling rotation speed)
    SUM(engagement_score) / 5.0 AS engagement_per_minute,
    
    -- Trend classification
    CASE 
        WHEN COUNT(*) >= 3 AND SUM(engagement_score) > 500 THEN 'VIRAL'
        WHEN COUNT(*) >= 2 AND SUM(engagement_score) > 200 THEN 'TRENDING'
        ELSE 'STABLE'
    END AS trend_status
    
FROM social_posts

GROUP BY 
    DATE_TRUNC('HOUR', posted_at),
    FLOOR(MINUTE(posted_at) / 5),
    hashtag

ORDER BY trend_window, total_engagement DESC;
```

**Workflow Explanation**:
1. **DATE_TRUNC + FLOOR** - Creates 5-minute segments (0-4, 5-9, 10-14 minutes)
2. **Window Label** - Combines hour and minute segment for readable timestamp
3. **Velocity Metrics** - Measures "momentum" within each fixed window (like measuring rotation speed)
4. **Trend Classification** - Each window evaluated independently—trend status resets every 5 minutes

## Use Case 7: Manufacturing Quality Control

**Scenario**: Monitor product defects in 30-minute tumbling windows on assembly lines.

```sql
CREATE OR REPLACE TABLE quality_checks (
    line_id VARCHAR(20),
    product_id VARCHAR(30),
    defect_type VARCHAR(30),
    severity_score INTEGER, -- 1-10, 10 = most severe
    checked_at TIMESTAMP
);

INSERT INTO quality_checks VALUES
('LINE_A', 'PROD_001', 'none', 0, '2024-08-01 08:00:15'),
('LINE_A', 'PROD_002', 'surface_scratch', 3, '2024-08-01 08:05:20'),
('LINE_A', 'PROD_003', 'dimension_off', 7, '2024-08-01 08:15:45'),
('LINE_A', 'PROD_004', 'none', 0, '2024-08-01 08:25:10'),
('LINE_A', 'PROD_005', 'color_mismatch', 4, '2024-08-01 08:35:30'), -- Next window
('LINE_B', 'PROD_101', 'none', 0, '2024-08-01 08:10:05');

-- TUMBLING WINDOW: Quality control analysis
SELECT 
    -- 30-minute tumbling windows (fixed, non-overlapping intervals)
    DATE_TRUNC('HOUR', checked_at) AS hour_start,
    CASE 
        WHEN MINUTE(checked_at) < 30 THEN '00-30'
        ELSE '30-60'
    END AS half_hour_segment,
    
    -- Complete window identifier
    DATEADD(MINUTE, 
        CASE WHEN MINUTE(checked_at) < 30 THEN 0 ELSE 30 END,
        DATE_TRUNC('HOUR', checked_at)) AS quality_window,
    
    line_id,
    
    -- Quality metrics (control over production process like controlling tumbling axis)
    COUNT(*) AS total_inspected,
    COUNT(CASE WHEN defect_type = 'none' THEN 1 END) AS passed,
    COUNT(CASE WHEN defect_type != 'none' THEN 1 END) AS defects_found,
    
    -- Defect rate (stability metric)
    COUNT(CASE WHEN defect_type != 'none' THEN 1 END) * 100.0 / COUNT(*) AS defect_rate_percent,
    
    -- Severity analysis
    AVG(CASE WHEN defect_type != 'none' THEN severity_score END) AS avg_defect_severity,
    MAX(severity_score) AS max_severity,
    
    -- Quality grade (process control rating)
    CASE 
        WHEN COUNT(CASE WHEN defect_type != 'none' THEN 1 END) * 100.0 / COUNT(*) < 5 
             AND MAX(severity_score) <= 5 THEN 'GRADE_A'
        WHEN COUNT(CASE WHEN defect_type != 'none' THEN 1 END) * 100.0 / COUNT(*) < 10 
             AND MAX(severity_score) <= 7 THEN 'GRADE_B'
        ELSE 'GRADE_C_NEEDS_ATTENTION'
    END AS quality_grade
    
FROM quality_checks

GROUP BY 
    DATE_TRUNC('HOUR', checked_at),
    CASE WHEN MINUTE(checked_at) < 30 THEN '00-30' ELSE '30-60' END,
    line_id

ORDER BY quality_window, line_id;
```

**Workflow Explanation**:
1. **DATE_TRUNC + CASE** - Splits each hour into two 30-minute tumbling windows
2. **Fixed Boundaries** - 08:00:00-08:30:00 and 08:30:00-09:00:00 are completely separate
3. **Control Metrics** - Defect rate acts like "axis control"—measures process stability
4. **Independent Grading** - Each window gets its own quality grade—problems in one window don't affect the next

## Key Tumbling Window Concepts Summary

| Concept | Gymnastics Analogy | SQL Implementation |
|---------|-------------------|-------------------|
| **Fixed Size** | Each tumbling pass has set length | `TIME_SLICE`, `DATE_TRUNC` with fixed intervals |
| **Non-overlapping** | Passes don't intersect | Each event belongs to exactly one window |
| **Complete Windows** | Full rotation from start to finish | Every window has defined start and end |
| **Axis Control** | Maintaining rotation stability | Using `GROUP BY` to maintain data boundaries |
| **External Factors** | Friction, gravity affecting motion | `WHERE` clauses filtering window data |

**Remember**: Tumbling windows in SQL are like gymnastics tumbling passes—each is a complete, independent sequence with clear boundaries. The data in one window never overlaps with another, just like each tumbling pass is distinct and separate!

> [!Tip]
> To effectively apply tumbling, try breaking down complex movements into simpler, manageable components and focus on controlling the axis of rotation.

## Key Points
- Tumbling involves rotation around a central axis, which can be fixed or moving.
- The speed and direction of the rotation can significantly affect the outcome of the tumbling process.
- External factors such as friction, gravity, and air resistance can influence tumbling motions.
- In gymnastics, tumbling passes are sequences of acrobatic skills performed in a fluid motion.

> [!important]
> The fundamental rule in tumbling is maintaining control over the axis of rotation to achieve the desired outcome without losing stability or safety.

## Mental Model
- Step 1: Identify the central axis of rotation.
- Step 2: Apply force or motion to initiate rotation around the axis.
- Step 3: Control the speed and direction of rotation to achieve the desired effect or outcome.

## Observe
- A common mistake people make is underestimating the impact of external factors on tumbling motions.
- A silent failure mode in tumbling is the gradual loss of control over the axis of rotation, leading to unpredictable outcomes.
- A misleading assumption is that tumbling is solely about the speed of rotation, ignoring the importance of control and direction.

> [!Important]
> In a gymnastics competition, a gymnast performs a tumbling pass, starting with a round-off, followed by a back handspring, and ending with a layout, demonstrating control over the axis of rotation to successfully land the skills.
