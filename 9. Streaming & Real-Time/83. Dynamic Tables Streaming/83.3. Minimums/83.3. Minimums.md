# 83.3. Minimums

>[!Note]
> Minimum requirements are the least acceptable standards that must be met to ensure quality, safety, or compliance, and understanding them is crucial for effective planning and decision-making.


Here are **6 use cases** with beginner-friendly workflows.

---

## Use Case 1: Clinical Trial Diversity Minimums (FDA Mandates)  
**The Minimum Concept:** FDA requires minimum representation of racial/ethnic minorities in clinical trials (e.g., at least 15% Hispanic/Latino participants for diabetes drugs). This is a **hard floor**, not a "nice to have."

```sql
WITH enrollment_tracking AS (
    SELECT 
        study_id,
        site_id,
        patient_id,
        race,
        ethnicity,
        enrollment_date,
        -- Define minimum requirements (regulatory floor)
        CASE 
            WHEN ethnicity = 'Hispanic or Latino' THEN 1 
            ELSE 0 
        END as is_hispanic,
        -- Running totals (monitoring compliance in real-time)
        SUM(is_hispanic) OVER (
            PARTITION BY study_id 
            ORDER BY enrollment_date 
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as cumulative_hispanic_enrolled,
        COUNT(*) OVER (
            PARTITION BY study_id 
            ORDER BY enrollment_date 
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as cumulative_total_enrolled,
        -- Calculate current percentage (measurable metric)
        (cumulative_hispanic_enrolled / cumulative_total_enrolled) * 100 as current_hispanic_pct,
        -- Define the MINIMUM THRESHOLD (the floor that cannot be violated)
        15.0 as regulatory_minimum_pct
    FROM pharma.clinical_enrollment
    WHERE study_id = 'DIAB-2026-PH3'
      AND enrollment_status = 'Randomized'
),
compliance_status AS (
    SELECT 
        study_id,
        enrollment_date,
        cumulative_total_enrolled as total_n,
        current_hispanic_pct,
        regulatory_minimum_pct,
        -- Gap analysis: How far above/below minimum?
        current_hispanic_pct - regulatory_minimum_pct as compliance_margin,
        -- Projection: At current enrollment rate, will we meet minimum at target N=300?
        (cumulative_hispanic_enrolled + 
         (300 - cumulative_total_enrolled) * 0.20) / 300 * 100 as projected_final_pct,
        CASE 
            WHEN current_hispanic_pct < regulatory_minimum_pct 
            THEN 'VIOLATION: Below regulatory floor'
            WHEN current_hispanic_pct < regulatory_minimum_pct + 5 
            THEN 'WARNING: Within 5% of minimum (buffer zone)'
            WHEN projected_final_pct < regulatory_minimum_pct 
            THEN 'SILENT FAILURE: Trending toward non-compliance at study end'
            ELSE 'COMPLIANT: Above minimum requirements'
        END as compliance_alert
    FROM enrollment_tracking
)
SELECT 
    study_id,
    total_n,
    ROUND(current_hispanic_pct, 2) || '%' as current_representation,
    regulatory_minimum_pct || '%' as fda_minimum,
    ROUND(compliance_margin, 2) || '%' as buffer_above_minimum,
    compliance_alert,
    CASE 
        WHEN compliance_alert LIKE 'VIOLATION%' 
        THEN 'HALT ENROLLMENT: Increase Hispanic/Latino recruitment immediately'
        WHEN compliance_alert LIKE 'SILENT FAILURE%' 
        THEN 'ACTION REQUIRED: Current sites underperforming, activate high-diversity sites'
        ELSE 'Continue monitoring'
    END as required_action
FROM compliance_status
WHERE total_n % 10 = 0  -- Check every 10 enrollments to avoid alert fatigue
   OR compliance_alert LIKE 'VIOLATION%'
   OR compliance_alert LIKE 'SILENT FAILURE%'
ORDER BY 
    total_n DESC;
```

### Detailed Workflow (Define → Measure → Enforce):

**Step 1: Defining the Measurable Minimum**
- We explicitly set `regulatory_minimum_pct = 15.0`—a hard number derived from FDA guidance.
- **Beginner Note:** This isn't a "goal" or "target" (which might be 25% for best practices). This is the **floor**—the study is invalid if it finishes below this.

**Step 2: Real-Time Compliance Calculation**
- Window functions calculate running percentages as each patient enrolls.
- **Why this matters:** We don't wait until study end to discover failure. The `cumulative_hispanic_enrolled` updates with every new row.

**Step 3: Distinguishing Minimum from Best Practice**
- The query checks `regulatory_minimum_pct + 5` to create a "buffer zone."
- **Common Mistake Prevented:** Teams often think "15% minimum means 15% is good." In reality, 15.1% is legally compliant but statistically risky (dropouts could push you below). The buffer identifies when you're technically compliant but operationally vulnerable.

**Step 4: Silent Failure Detection (Trending Violation)**
- `projected_final_pct` calculates: "If we enroll the remaining 100 patients at the current Hispanic recruitment rate (20%), where do we finish?"
- **Silent Failure:** A study at 16% Hispanic enrollment with 100 patients looks compliant, but if those 100 are all from low-diversity sites, the final study might end at 14%. This catches the trajectory before it becomes a violation.

**Step 5: Enforcement Logic**
- The `required_action` column translates statistical data into operational directives.
- **Measurable & Achievable:** The system tells you exactly how many more Hispanic patients needed: `CEIL((0.15 * 300) - current_hispanic_enrolled)`.

## Use Case 2: Cold Chain Temperature Minimums (Stability Thresholds)  
**The Minimum Concept:** Drugs like insulin require storage between 2°C-8°C. Below 2°C is a **violation** (freezing denatures proteins), but 2°C is the minimum acceptable—distinct from the "optimal" 4-5°C target.

```sql
WITH temperature_readings AS (
    SELECT 
        batch_id,
        storage_unit_id,
        sensor_timestamp,
        temperature_celsius,
        -- Define the HARD FLOOR (violation boundary)
        2.0 as min_acceptable_temp,
        8.0 as max_acceptable_temp,
        -- Define the OPTIMAL TARGET (best practice, not minimum)
        5.0 as optimal_target_temp,
        -- Detect excursions below minimum (the violation)
        CASE 
            WHEN temperature_celsius < 2.0 THEN 'CRITICAL: Below minimum storage temp'
            WHEN temperature_celsius > 8.0 THEN 'CRITICAL: Above maximum storage temp'
            ELSE 'Within range'
        END as temp_status,
        -- Calculate deviation from minimum (how close to the edge?)
        temperature_celsius - 2.0 as degrees_above_minimum,
        -- Time-weighted violation severity (duration matters)
        LAG(sensor_timestamp) OVER (
            PARTITION BY batch_id, storage_unit_id 
            ORDER BY sensor_timestamp
        ) as previous_reading_time
    FROM pharma.cold_chain_monitoring
    WHERE sensor_timestamp >= CURRENT_DATE - 7
),
excursion_duration AS (
    SELECT 
        batch_id,
        storage_unit_id,
        sensor_timestamp,
        temperature_celsius,
        min_acceptable_temp,
        degrees_above_minimum,
        temp_status,
        -- Calculate time spent in violation (continuous duration)
        SUM(CASE WHEN temp_status LIKE 'CRITICAL%' THEN 1 ELSE 0 END) OVER (
            PARTITION BY batch_id, storage_unit_id, 
            CASE WHEN temp_status LIKE 'CRITICAL%' THEN 1 ELSE 0 END
            ORDER BY sensor_timestamp
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as violation_duration_minutes,
        -- Regulatory minimum: Any excursion >30 minutes = batch destruction
        CASE 
            WHEN temp_status LIKE 'CRITICAL%' 
                 AND DATEDIFF(minute, previous_reading_time, sensor_timestamp) > 30 
            THEN 'BATCH COMPROMISED: Exceeds 30-min maximum excursion limit'
            WHEN degrees_above_minimum < 1.0 AND temp_status = 'Within range' 
            THEN 'BUFFER WARNING: Within 1°C of minimum floor'
            ELSE 'Stable'
        END as batch_disposition
    FROM temperature_readings
)
SELECT 
    batch_id,
    storage_unit_id,
    MIN(temperature_celsius) as lowest_recorded_temp,
    COUNT(CASE WHEN temp_status LIKE 'CRITICAL%' THEN 1 END) as violation_reading_count,
    MAX(CASE 
        WHEN batch_disposition = 'BATCH COMPROMISED: Exceeds 30-min maximum excursion limit' 
        THEN 1 ELSE 0 
    END) as is_batch_destroyed,
    AVG(degrees_above_minimum) as avg_safety_margin,
    -- The "Minimum Requirements" compliance report
    CASE 
        WHEN is_batch_destroyed = 1 THEN 'NON-COMPLIANT: Batch must be destroyed per SOP'
        WHEN MIN(temperature_celsius) >= 2.0 THEN 'COMPLIANT: Never breached minimum threshold'
        ELSE 'INVESTIGATE: Brief excursion but within time limits'
    END as regulatory_compliance_status
FROM excursion_duration
GROUP BY 
    batch_id, storage_unit_id
HAVING 
    MIN(temperature_celsius) < 8.0  -- Only show batches that approached limits
ORDER BY 
    lowest_recorded_temp ASC;
```

### Detailed Workflow (Defining the Floor vs. the Target):

**Step 1: Explicit Threshold Definition**
- We define three numbers: `min_acceptable_temp` (2°C), `max_acceptable_temp` (8°C), and `optimal_target_temp` (5°C).
- **Theory Connection:** This demonstrates the difference between "minimum requirements" (2°C) and "best practices" (5°C). A batch stored at 2.5°C is compliant but suboptimal—it meets the legal floor but not the quality target.

**Step 2: Continuous Violation Tracking**
- The query uses window functions to calculate how long a batch has been in violation state.
- **Measurable Standard:** FDA guidelines specify that brief excursions (<15 minutes) may be acceptable if stability data supports it, but >30 minutes is automatic destruction. The query enforces this measurable time limit.

**Step 3: Buffer Zone Monitoring**
- `degrees_above_minimum` tracks how close we are to the 2°C floor during normal operation.
- **Silent Failure:** A freezer cycling between 2.1°C and 2.5°C is technically compliant but indicates equipment failure imminent. The "BUFFER WARNING" catches this before it becomes a violation.

**Step 4: Disposition Logic**
- The `CASE` statement automates the regulatory decision: destroy or release.
- **Achievable Standard:** The rule is binary and clear—no ambiguity about whether a batch is usable.

## Use Case 3: Manufacturing Batch Yield Minimums (Efficiency Floors)  
**The Minimum Concept:** While the goal is 95% yield, the **minimum acceptable yield** is 85% (below which the batch is unprofitable and investigational). This is a business-critical floor, not an aspiration.

```sql
WITH batch_production AS (
    SELECT 
        batch_id,
        product_code,
        manufacturing_date,
        theoretical_yield_units,
        actual_yield_units,
        -- Calculate the yield percentage
        (actual_yield_units / theoretical_yield_units) * 100 as actual_yield_pct,
        -- Define minimum requirements (the floor)
        85.0 as minimum_acceptable_yield_pct,
        -- Define optimal target (best practice)
        95.0 as target_yield_pct,
        -- Cost of goods sold calculation
        (theoretical_yield_units - actual_yield_units) * 12.50 as lost_revenue_usd,
        -- Identify which stage caused loss (yield loss attribution)
        raw_material_waste_units,
        processing_loss_units,
        packaging_loss_units
    FROM pharma.batch_records
    WHERE manufacturing_date >= '2026-01-01'
),
yield_analysis AS (
    SELECT 
        batch_id,
        product_code,
        actual_yield_pct,
        minimum_acceptable_yield_pct,
        target_yield_pct,
        -- Gap analysis
        actual_yield_pct - minimum_acceptable_yield_pct as margin_above_minimum,
        target_yield_pct - actual_yield_pct as gap_to_target,
        lost_revenue_usd,
        -- Regulatory/Business rule: Below 85% triggers CAPA (Corrective Action)
        CASE 
            WHEN actual_yield_pct < minimum_acceptable_yield_pct 
            THEN 'VIOLATION: Batch below minimum yield - Initiate investigation'
            WHEN actual_yield_pct < minimum_acceptable_yield_pct + 2 
            THEN 'WARNING: Within 2% of minimum floor - Monitor closely'
            WHEN actual_yield_pct >= target_yield_pct 
            THEN 'EXCEEDS TARGET: Best practice performance'
            ELSE 'ACCEPTABLE: Above minimum, below target'
        END as yield_classification,
        -- Identify primary loss source for optimization
        GREATEST(raw_material_waste_units, processing_loss_units, packaging_loss_units) as primary_loss_source
    FROM batch_production
),
trend_analysis AS (
    SELECT 
        product_code,
        COUNT(*) as batch_count,
        AVG(actual_yield_pct) as avg_yield,
        MIN(actual_yield_pct) as worst_yield,
        STDDEV(actual_yield_pct) as yield_variability,
        -- Silent failure: Gradual decline toward minimum
        REGR_SLOPE(actual_yield_pct, DATEDIFF(day, '2026-01-01', manufacturing_date)) as yield_trend_per_day,
        SUM(CASE WHEN actual_yield_pct < minimum_acceptable_yield_pct THEN 1 ELSE 0 END) as subminimum_batch_count
    FROM batch_production
    JOIN yield_analysis USING (batch_id)
    GROUP BY 
        product_code
)
SELECT 
    ya.batch_id,
    ya.product_code,
    ROUND(ya.actual_yield_pct, 2) || '%' as batch_yield,
    ya.minimum_acceptable_yield_pct || '%' as floor,
    ya.target_yield_pct || '%' as target,
    ya.yield_classification,
    '$' || ROUND(ya.lost_revenue_usd, 2) as financial_impact,
    ta.yield_trend_per_day,
    CASE 
        WHEN ta.yield_trend_per_day < -0.05 
        THEN 'SILENT FAILURE: Process drifting toward minimum over time'
        WHEN ya.actual_yield_pct < ya.minimum_acceptable_yield_pct 
        THEN 'IMMEDIATE ACTION: Quarantine batch, initiate CAPA'
        ELSE 'Continue monitoring'
    END as operational_decision
FROM yield_analysis ya
JOIN trend_analysis ta ON ya.product_code = ta.product_code
ORDER BY 
    ya.actual_yield_pct ASC;
```

### Detailed Workflow (Business-Critical Floors):

**Step 1: Dual Threshold System**
- We establish both `minimum_acceptable_yield_pct` (85%) and `target_yield_pct` (95%).
- **Key Distinction:** A batch at 87% passes release (meets minimum) but triggers operational review (far from target). This prevents the "common mistake" of treating minimum as optimal.

**Step 2: Financial Translation**
- `lost_revenue_usd` converts percentage points into dollars—making the abstract "yield" tangible for stakeholders.
- **Measurable Impact:** Leadership understands "$15,000 lost revenue" better than "2% yield loss."

**Step 3: Trend-Based Silent Failure**
- `REGR_SLOPE` calculates if yields are gradually declining (e.g., losing 0.1% per day).
- **Silent Failure Mode:** Individual batches might pass (88%, 87%, 86.5%), but the trend shows you'll hit the 85% floor in weeks. This catches the "gradual increase in lag" equivalent for manufacturing.

**Step 4: Root Cause Attribution**
- `GREATEST()` identifies which waste stream (raw materials, processing, packaging) is the bottleneck.
- **Achievable Correction:** Instead of "improve everything," the query tells you exactly which process step needs investment.

## Use Case 4: Drug Potency Stability Minimums (Shelf Life)  
**The Minimum Concept:** A drug must maintain ≥90% potency throughout its shelf life. Below 90% at expiry = failed batch, but we also monitor to ensure we don't approach that floor during normal aging.

```sql
WITH stability_testing AS (
    SELECT 
        batch_id,
        storage_condition,  -- 'Accelerated', 'Long-term', 'Stress'
        test_month,         -- 0, 3, 6, 12, 24, 36 months
        potency_assay_result,  -- Percentage of label claim
        -- The REGULATORY FLOOR (cannot ship below this)
        90.0 as minimum_potency_pct,
        -- The WARNING THRESHOLD (internal quality standard)
        95.0 as alert_threshold_pct,
        -- Calculate degradation rate (slope of potency loss)
        (potency_assay_result - LAG(potency_assay_result) OVER (
            PARTITION BY batch_id, storage_condition 
            ORDER BY test_month
        )) / (test_month - LAG(test_month) OVER (
            PARTITION BY batch_id, storage_condition 
            ORDER BY test_month
        )) as potency_loss_per_month
    FROM pharma.stability_studies
    WHERE test_month IS NOT NULL
),
shelf_life_projection AS (
    SELECT 
        batch_id,
        storage_condition,
        test_month,
        potency_assay_result,
        minimum_potency_pct,
        potency_loss_per_month,
        -- Project when potency hits minimum (extrapolation)
        CASE 
            WHEN potency_loss_per_month < 0  -- Only if degrading
            THEN (potency_assay_result - minimum_potency_pct) / ABS(potency_loss_per_month)
            ELSE NULL  -- Stable or increasing
        END as months_until_failure,
        -- Current status vs minimum
        potency_assay_result - minimum_potency_pct as current_margin,
        CASE 
            WHEN potency_assay_result < minimum_potency_pct 
            THEN 'FAILED: Below minimum potency - Do not release'
            WHEN potency_assay_result < alert_threshold_pct 
            THEN 'ALERT: Approaching minimum (within 5%)'
            WHEN months_until_failure < 6 AND months_until_failure IS NOT NULL
            THEN 'EXPIRY WARNING: Will hit minimum within 6 months'
            ELSE 'ACCEPTABLE: Robust margin above minimum'
        END as stability_status
    FROM stability_testing
),
regulatory_reporting AS (
    SELECT 
        batch_id,
        MAX(CASE WHEN storage_condition = 'Long-term' AND test_month = 0 THEN potency_assay_result END) as initial_potency,
        MAX(CASE WHEN storage_condition = 'Long-term' AND test_month = 24 THEN potency_assay_result END) as month_24_potency,
        MAX(CASE WHEN storage_condition = 'Accelerated' AND test_month = 6 THEN potency_assay_result END) as accelerated_6month,
        -- ICH Guidelines: If accelerated drops below 90% at 6 months, long-term is suspect
        CASE 
            WHEN MAX(CASE WHEN storage_condition = 'Accelerated' AND test_month = 6 THEN potency_assay_result END) < 90 
            THEN 'FAILED ICH: Accelerated stability below minimum'
            ELSE 'ICH Compliant'
        END as ich_q1a_status
    FROM stability_testing
    GROUP BY 
        batch_id
)
SELECT 
    sp.batch_id,
    sp.storage_condition,
    sp.test_month,
    sp.potency_assay_result || '%' as current_potency,
    sp.minimum_potency_pct || '%' as regulatory_minimum,
    ROUND(sp.current_margin, 2) || '%' as safety_margin,
    ROUND(sp.months_until_failure, 1) as projected_months_to_failure,
    sp.stability_status,
    rr.ich_q1a_status,
    -- Final disposition
    CASE 
        WHEN sp.stability_status LIKE 'FAILED%' OR rr.ich_q1a_status LIKE 'FAILED%' 
        THEN 'QUARANTINE: Destroy batch or rework'
        WHEN sp.months_until_failure < 12 
        THEN 'SHORTEN SHELF LIFE: Reduce expiry to ' || FLOOR(sp.test_month + sp.months_until_failure) || ' months'
        ELSE 'APPROVE: Standard 36-month expiry'
    END as batch_disposition
FROM shelf_life_projection sp
LEFT JOIN regulatory_reporting rr ON sp.batch_id = rr.batch_id
WHERE sp.test_month = (SELECT MAX(test_month) FROM stability_testing st2 WHERE st2.batch_id = sp.batch_id)
ORDER BY 
    sp.current_margin ASC;
```

### Detailed Workflow (Degradation Curves and Floors):

**Step 1: The Absolute Floor**
- `minimum_potency_pct = 90` is the ICH Q1A regulatory requirement. This is a **binary pass/fail** at expiry.

**Step 2: The Projection Window**
- We calculate `months_until_failure` using linear regression of degradation rate.
- **Proactive Compliance:** Instead of waiting 36 months to discover failure, we project at month 6 whether the trend will hit the floor by month 36.

**Step 3: Dual Condition Monitoring**
- We compare `Long-term` (real-time) vs `Accelerated` (stress testing) conditions.
- **Silent Failure:** Sometimes long-term looks good (95% at 24 months) but accelerated testing shows rapid degradation (85% at 6 months), indicating the formulation is unstable. The query catches this mismatch.

**Step 4: Margin Management**
- `current_margin` tracks how far above 90% we are.
- **Best Practice vs Minimum:** A batch at 91% is compliant but risky (measurement variability could make it 89% in reality). We want batches staying above 95% (the alert threshold) to ensure we never accidentally ship sub-potent product.

## Use Case 5: Pharmacovigilance Reporting Time Minimums (Regulatory Deadlines)  
**The Minimum Concept:** FDA requires expedited reporting of serious adverse events (SAEs) within 15 calendar days. This is a **maximum time** (ceiling), but analytically we treat it as a "minimum speed requirement"—you must report at least as fast as 15 days.

```sql
WITH ae_reports AS (
    SELECT 
        case_id,
        seriousness_criteria,  -- 'Death', 'Life-threatening', 'Hospitalization', etc.
        event_onset_date,
        sponsor_aware_date,    -- Day 0 for regulatory clock
        report_submitted_date, -- When FDA received report
        -- Calculate the reporting lag (duration)
        DATEDIFF(day, sponsor_aware_date, report_submitted_date) as days_to_report,
        -- Define the REGULATORY MAXIMUM (ceiling becomes floor for compliance)
        15 as maximum_allowable_days,  -- Minimum speed = 15 days max
        -- Define the INTERNAL TARGET (best practice)
        7 as target_reporting_days,
        -- Business hours calculation (excluding weekends for metric accuracy)
        5 * (DATEDIFF(week, sponsor_aware_date, report_submitted_date)) 
            + CASE WHEN DAYOFWEEK(sponsor_aware_date) = 1 THEN 6 
                   WHEN DAYOFWEEK(sponsor_aware_date) = 7 THEN 5 
                   ELSE 7 - DAYOFWEEK(sponsor_aware_date) END
            + CASE WHEN DAYOFWEEK(report_submitted_date) = 1 THEN 0 
                   WHEN DAYOFWEEK(report_submitted_date) = 7 THEN 0 
                   ELSE DAYOFWEEK(report_submitted_date) - 1 END as business_days_to_report
    FROM pharma.safety_reports
    WHERE seriousness_criteria IN ('Death', 'Life-threatening')
      AND sponsor_aware_date >= CURRENT_DATE - 90
),
compliance_calculation AS (
    SELECT 
        case_id,
        seriousness_criteria,
        sponsor_aware_date,
        days_to_report,
        business_days_to_report,
        maximum_allowable_days,
        target_reporting_days,
        -- Days remaining until deadline (negative = overdue)
        maximum_allowable_days - days_to_report as days_remaining,
        -- Compliance categories
        CASE 
            WHEN days_to_report > maximum_allowable_days 
            THEN 'VIOLATION: Late reporting (>15 days) - FDA reportable'
            WHEN days_to_report = maximum_allowable_days 
            THEN 'BORDERLINE: Exactly at limit - High risk'
            WHEN days_to_report <= target_reporting_days 
            THEN 'EXEMPLARY: Within best practice target'
            ELSE 'COMPLIANT: Within regulatory limit but above target'
        END as compliance_category,
        -- Trend analysis: Are we getting slower?
        AVG(days_to_report) OVER (
            ORDER BY sponsor_aware_date 
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ) as rolling_7_case_average
    FROM ae_reports
),
predictive_alert AS (
    SELECT 
        cc.*,
        -- If current open cases follow trend, will they miss deadline?
        CASE 
            WHEN rolling_7_case_average > 10 AND days_to_report IS NULL 
            THEN 'AT RISK: Open case may exceed 15-day limit based on recent performance'
            ELSE 'On track'
        END as open_case_risk,
        -- Cumulative compliance rate
        SUM(CASE WHEN days_to_report <= maximum_allowable_days THEN 1 ELSE 0 END) OVER (
            ORDER BY sponsor_aware_date
        ) / ROW_NUMBER() OVER (ORDER BY sponsor_aware_date) * 100 as cumulative_compliance_rate
    FROM compliance_calculation cc
)
SELECT 
    case_id,
    seriousness_criteria,
    sponsor_aware_date,
    days_to_report || ' days' as actual_duration,
    maximum_allowable_days || ' days' as regulatory_limit,
    target_reporting_days || ' days' as internal_target,
    days_remaining || ' days' as buffer_remaining,
    compliance_category,
    ROUND(cumulative_compliance_rate, 1) || '%' as ytd_compliance_rate,
    CASE 
        WHEN compliance_category LIKE 'VIOLATION%' 
        THEN 'IMMEDIATE: File FDA deviation report, implement CAPA'
        WHEN compliance_category = 'BORDERLINE' 
        THEN 'URGENT: Process improvement required - too close to limit'
        WHEN rolling_7_case_average > 12 
        THEN 'TREND ALERT: Process slowing, nearing systematic failure'
        ELSE 'Monitor'
    END as management_action
FROM predictive_alert
WHERE days_to_report IS NOT NULL  -- Only closed cases
   OR open_case_risk = 'AT RISK: Open case may exceed 15-day limit based on recent performance'
ORDER BY 
    days_to_report DESC;
```

### Detailed Workflow (Time-Critical Minimums):

**Step 1: Inverting the Concept**
- While 15 days is a "maximum," we treat compliance as a binary minimum requirement: "You must complete reporting in ≤15 days."
- **Measurable:** The `days_to_report` metric is unambiguous.

**Step 2: Buffer Management**
- We calculate `days_remaining` (15 - actual). When this approaches 0, we're at the floor.
- **Silent Failure:** A department averaging 14.5 days is technically 100% compliant but one weekend delay creates a violation. The `rolling_7_case_average` detects this operational brittleness.

**Step 3: Business vs Calendar Days**
- The complex `business_days_to_report` calculation excludes weekends.
- **Regulatory Nuance:** FDA counts calendar days, but business days show operational capacity. A 15-day calendar report that spans 2 weekends (11 business days) is compliant but inefficient.

**Step 4: Predictive Compliance**
- For open cases (not yet reported), we compare days elapsed against the recent trend.
- **Proactive Enforcement:** If recent cases took 12 days average, and an open case is already 10 days old, we flag it as "at risk" before it becomes a violation.

## Use Case 6: Laboratory Reference Range Minimums (Clinical Thresholds)  
**The Minimum Concept:** For safety labs, values below the Lower Limit of Normal (LLN) trigger dose holds. This is a **clinical floor** for patient safety, distinct from the "optimal" healthy range.

```sql
WITH lab_results AS (
    SELECT 
        patient_id,
        visit_date,
        test_name,
        result_value,
        unit_of_measure,
        -- Population reference ranges (the minimums for safety)
        CASE test_name
            WHEN 'Hemoglobin' THEN 12.0  -- g/dL (LLN for females)
            WHEN 'Absolute Neutrophil Count' THEN 1.5  -- K/uL
            WHEN 'Platelet Count' THEN 100  -- K/uL
            WHEN 'Creatinine' THEN 0.6  -- mg/dL (lower minimum, but upper matters more)
        END as lower_limit_normal,
        CASE test_name
            WHEN 'Hemoglobin' THEN 16.0  -- Upper limit
            WHEN 'Absolute Neutrophil Count' THEN 8.0
            WHEN 'Platelet Count' THEN 400
            WHEN 'Creatinine' THEN 1.2
        END as upper_limit_normal,
        -- Calculate where result sits relative to minimum
        result_value - lower_limit_normal as distance_from_minimum,
        (result_value / lower_limit_normal) * 100 as pct_of_minimum,
        -- Dose-limiting toxicity (DLT) thresholds (stricter than LLN)
        CASE test_name
            WHEN 'Hemoglobin' THEN 10.0
            WHEN 'Absolute Neutrophil Count' THEN 1.0
            WHEN 'Platelet Count' THEN 75
        END as dose_hold_minimum
    FROM pharma.laboratory_values
    WHERE test_name IN ('Hemoglobin', 'Absolute Neutrophil Count', 'Platelet Count', 'Creatinine')
      AND visit_date >= CURRENT_DATE - 30
),
safety_assessment AS (
    SELECT 
        patient_id,
        visit_date,
        test_name,
        result_value,
        lower_limit_normal,
        pct_of_minimum,
        -- Clinical decision logic based on minimum thresholds
        CASE 
            WHEN result_value < dose_hold_minimum 
            THEN 'DOSE HOLD: Below DLT minimum - Stop study drug'
            WHEN result_value < lower_limit_normal 
            THEN 'BELOW LLN: Clinically significant - Monitor weekly'
            WHEN result_value < lower_limit_normal * 1.1 
            THEN 'WARNING: Within 10% of lower limit'
            ELSE 'NORMAL: Above minimum threshold'
        END as clinical_action,
        -- Track trends (are we approaching the minimum?)
        LAG(result_value) OVER (
            PARTITION BY patient_id, test_name 
            ORDER BY visit_date
        ) as previous_result,
        result_value - LAG(result_value) OVER (
            PARTITION BY patient_id, test_name 
            ORDER BY visit_date
        ) as change_from_last
    FROM lab_results
)
SELECT 
    patient_id,
    test_name,
    result_value || ' ' || unit_of_measure as current_value,
    lower_limit_normal || ' ' || unit_of_measure as minimum_safe,
    ROUND(pct_of_minimum, 1) || '%' as percent_of_minimum,
    clinical_action,
    change_from_last,
    -- Silent failure: Gradual decline toward minimum
    CASE 
        WHEN change_from_last < 0 
             AND result_value < lower_limit_normal * 1.2 
             AND result_value > lower_limit_normal 
        THEN 'SILENT DECLINE: Trending toward minimum but still "normal"'
        WHEN clinical_action LIKE 'DOSE HOLD%' 
        THEN 'INTERVENTION REQUIRED'
        ELSE 'Stable'
    END as trend_status,
    -- Multi-parameter safety check (worst value drives decision)
    MAX(CASE WHEN clinical_action LIKE 'DOSE HOLD%' THEN 1 ELSE 0 END) OVER (
        PARTITION BY patient_id, visit_date
    ) as any_dose_hold_triggered
FROM safety_assessment
ORDER BY 
    pct_of_minimum ASC;
```

### Detailed Workflow (Clinical Floors for Patient Safety):

**Step 1: Dual Minimum System**
- We define both `lower_limit_normal` (population reference) and `dose_hold_minimum` (stricter study-specific).
- **Distinction:** A hemoglobin of 11.5 is below normal (anemic) but might allow continued treatment, while <10.0 requires stopping the drug immediately. The query enforces the stricter clinical minimum.

**Step 2: Percentage of Minimum**
- `pct_of_minimum` shows how close to the cliff edge we are. A value at 105% of minimum has only 5% buffer.
- **Measurable:** Percentages normalize across different tests (platelets vs hemoglobin).

**Step 3: Trend-Based Silent Failure**
- The query detects when a patient is declining toward the minimum but still technically above it.
- **Clinical Relevance:** Catching a drop from 15 → 12.5 → 11 hemoglobin over three visits allows intervention before hitting the 10.0 dose-hold floor.

**Step 4: Cross-Panel Aggregation**
- `any_dose_hold_triggered` uses a window function to check across all lab values for that patient on that date.
- **Systemic View:** Even if platelets are fine, if neutrophils triggered a dose hold, the whole patient is flagged for drug hold.
