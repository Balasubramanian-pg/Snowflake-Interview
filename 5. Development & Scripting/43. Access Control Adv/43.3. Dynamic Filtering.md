# Dynamic Filtering in Snowflake

**Definition**

Runtime data restriction based on session context, user attributes, or external variables without hardcoding filters in queries or duplicating views per user segment.

**Applications (SQL Queries)**

Context function approach:
```sql
SELECT * FROM sales 
WHERE region = CURRENT_REGION();
```

Session variable filtering:
```sql
ALTER SESSION SET team = 'finance';
SELECT * FROM projects WHERE department = $team;
```

Secure UDF with lookup:
```sql
CREATE FUNCTION user_allowed_customers() 
RETURNS TABLE (customer_id INT)
AS $$ 
  SELECT customer_id FROM permissions 
  WHERE user_name = CURRENT_USER()
$$;
```

Row access policy:
```sql
CREATE ROW ACCESS POLICY region_filter AS (region STRING) 
RETURNS BOOLEAN ->
region = CURRENT_REGION();
```

**Edge Cases**

- Context returns NULL: Unset variables bypass filters. Always validate `IS NOT NULL`.
- Case sensitivity: `CURRENT_REGION()` returns uppercase. Data may be mixed case. Use `UPPER()` or `COLLATE`.
- Performance degradation: Functions executing per row. Materialize allowed sets in temporary tables.
- Concurrent sessions: Variables persist per session. Connection pooling may leak contexts. Reset at session start.

**Common Pitfalls**

- Assuming context functions are secure. Users can override session variables in worksheets.
- Filtering in application layer only. Always enforce at database level with policies.
- Hardcoding business logic in policies. Externalize to configuration tables for maintenance.
- Ignoring result cache. Dynamic filters may return stale data if cache ignores context. Use `NO_CACHE` hint or disable for sensitive queries.
- Missing fallback handling. No ELSE clause in policy logic exposes all rows. Default to deny.
